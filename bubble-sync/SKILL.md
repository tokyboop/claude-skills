# Bubble Sync：对话精华提炼与同步

你的工作是充当对话的"记录员"——在用户喊出同步指令后，快速从整段对话中找出值得保留的东西，脱敏后存档，让知识在未来对话中继续发挥价值。

---

## 第一步：扫描对话，找出有价值的内容

通读整段对话，寻找以下类型的内容（跳过闲聊和已有记录的重复内容）：

- **工作流 / 操作步骤**：新学到的做事方法、命令序列、工具用法
- **踩坑 / 错误修复**：遇到的问题和解决方案，尤其是不明显的 root cause
- **业务规则 / 定义**：业务逻辑、字段含义、计算公式等确认事项
- **工具配置**：安装路径、环境变量、依赖关系等
- **决策结论**：用户明确拍板的选择（"我们用X不用Y，因为..."）
- **可复用流程**：如果某套步骤将来还会重复，就是潜在的 skill

如果对话主要是闲聊、已知内容的复述，或者没有新的可操作知识——直接告诉用户：
> "本次对话没有发现值得新增同步的内容，已有记忆已是最新状态。"

---

## 第二步：分类每条内容

对每条有价值的内容，判断它属于哪种类型：

| 类型 | 判断标准 | 写到哪里 |
|------|---------|---------|
| **memory** | 用户偏好、业务定义、项目背景、一次性配置 | `~/.claude/projects/.../memory/` |
| **skill** | 可复用的多步骤操作流程，将来会反复用 | `E:/ClaudeTask/claude-skills/<name>/SKILL.md` |
| **wiki** | 今日学习总结、实验记录、经验文章 | `E:/ClaudeTask/wiki_<YYYY-MM-DD>_<topic>.md` |

**判断原则**：
- 只会用一次的背景信息 → memory
- 有明确步骤、将来会重复的流程 → skill
- 综合性的今日总结、包含多个知识点 → wiki
- 可以同时写多种类型（一次对话可以同时产出 memory + wiki）

---

## 第三步：脱敏（写文件之前必须完成）

**脱敏的目的**：内容要推送 GitHub，不能含有任何个人或敏感信息。保留结构和规律，去掉具体敏感值。

替换规则：
- 用户名/路径中的个人标识 → `<USER>`（如 `/Users/guohuijun/` → `/Users/<USER>/`）
- API Key / Token / 密码 → `<REDACTED>`
- 真实 IP 地址 → `<IP>`
- 公司内部项目代号、数据库名、表名（如非公开）→ `<PROJECT>` / `<DB>` / `<TABLE>`
- 真实域名（非公开服务）→ `<INTERNAL_DOMAIN>`
- 具体数值型业务阈值（如充值金额分层）→ 保留结构，用注释标注"具体值见内部文档"

**不需要脱敏的内容**：公开工具名、命令语法、通用编程模式、开源项目名称。

脱敏完成后，在同步摘要里列出"脱敏了哪些内容"。

---

## 第四步：写文件

### Memory 文件格式
```markdown
---
name: <简短名称>
description: <一句话说明用途>
type: <user|feedback|project|reference>
---

<内容>
```
写完后更新 `MEMORY.md` 索引（追加一行指针）。

### Skill 文件
按标准 SKILL.md 格式写到 `E:/ClaudeTask/claude-skills/<name>/SKILL.md`，同时更新 `E:/ClaudeTask/claude-skills/README.md` 的技能列表。

### Wiki 文件
写到 `E:/ClaudeTask/wiki_<YYYY-MM-DD>_<topic>.md`，结构自由，但要包含：背景、核心内容、经验教训三个部分。

---

## 第五步：推送 GitHub

```bash
export PATH="$PATH:/c/Program Files/GitHub CLI"
cd /e/ClaudeTask/claude-skills
git add .
git commit -m "sync: <简短描述今日内容>"
git push
```

Wiki 文件默认不推送（存在本地即可），除非用户明确要求。

---

## 第六步：输出同步摘要

用下面的格式告诉用户做了什么：

---

### 泡泡同步完成

**提炼到的内容：**
- [memory] `<文件名>` — <一句话说明>
- [skill] `<skill名>` — <一句话说明>
- [wiki] `<文件名>` — <一句话说明>

**脱敏处理：**
- <具体脱敏了什么，例如：替换了3处路径中的用户名>

**GitHub 推送：**
- ✅ 已推送 / ⏭️ wiki 未推送（本地存档）

---

## 边界情况处理

- **内容已存在**：如果对话中的知识点已在 memory 里有记录，不要重复写，可以说"已有记录，无需更新"
- **不确定分类**：倾向写 wiki，wiki 包容性最强，之后可以拆分
- **对话很长**：重点关注后半段（用户通常在对话后期确认结论），前面的探索过程可以略去
- **用户只说"泡泡"没有"总结同步"**：判断上下文，如果像是在呼唤助手询问事情，不要误触发同步流程
